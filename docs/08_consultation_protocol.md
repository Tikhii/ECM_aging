# 在线咨询协议 (Consultation Protocol)

> 离线区脚本遇到无法自我修复的异常后, 如何以**最小必要信息**外带到在线 Claude
> 对话而不泄露敏感实验数据, 本文件给出约束与模板。配套 `docs/07_offline_runbook.md`
> 使用: 先按 runbook 自修复, 失败才按本协议外带。

---

## §0 适用范围

本协议规定跨 air-gap 的人机咨询通道。适用场景是: 离线工作站上
遇到涉及模型架构、物理机理、参数可识别性等需要深度讨论的疑难
问题, 需要外带抽象化的观测笔记与联网环境的 Claude 讨论。

本协议不适用于日常拟合操作中的常规故障处置。后者应查阅
docs/07_offline_runbook.md 中对应错误码条目的 remediation 字段,
在离线机上按步骤自行解决。

本协议的实际使用频率预期较低 (受限于数据流出的通道限制), 但
其存在是必要的——它定义了在必要时合法可用的唯一咨询通道,
以及该通道下合法可外带的信息边界。

---

## §1 为什么需要这份协议

本工程采用 **air-gapped 三阶段工作流**:

```
  原始数据 / 实验硬件 ─┐
                     │  (1) 离线区: Claude Code + 本地脚本, 仅读写本机文件
    docs/ 文档 ──────┤
                     │  (2) 外带区: 只有工程师手工贴给在线 Claude 的文本
   在线 Claude 对话 ─┘      能跨越离线↔在线的边界
```

离线区能看到全部原始数据与脚本; 在线区无法 (也不应) 看到原始数据。绝大多数故
障诊断应在离线区由 Claude Code + runbook 完成; 只有下列情况需要进入在线咨询:

- 遇到 runbook 里没登记的错误码以外的新故障模式
- 错误码登记了但 `remediation` 已穷尽仍未解决
- 拟合结果在物理合理性上明显可疑, 需要论文/方法学层面的讨论
- SOP / 架构级修订

这些场合都不需要绝对数值。**原始数据不应出现在在线对话里**, 无论格式如何、无论
脱敏程度如何。原因:

1. **数据反推**: 单一参数的绝对数值 + 电池型号即可用于反推供应商工艺; 多个
   cell 的散点可用于估计实验规模与 DOE。
2. **对话日志无限期缓存**: 网页端对话历史在服务端长期保留, 已发送的内容无法
   真正撤回。
3. **账号混用**: 同一账号在不同项目间切换时, 历史对话可能被后续项目的上下文
   读取。

这三点联合导致: 一旦绝对数值被贴出, 事实上就已经脱离了工程的边界。

## §2 可外带信息白名单 (positive list)

以下是**穷举列表**; 不在此列表内的, 默认不可外带。

1. **脚本名、版本号、git commit hash**
   形如 `scripts/fit_calendar.py @ git a955a45`。commit hash 可公开, 因为
   `.git/objects` 不含数据文件的明文。
2. **错误码**
   仅限 `docs/error_codes_registry.json` 中已登记的编号 (如 `FIT4A-E003`)。
   未登记的错误描述不可外带 —— 先登记再讨论。
3. **退出码与异常栈**
   异常栈必须去敏感路径: 去掉 `/Users/.../experiments/.../cell_E7.csv` 这种
   含 cell ID 的路径, 保留 Python 层的类型与行号。
4. **运行规模量级**
   允许: `N_cells ~ O(10)`, `N_RPT/cell ~ O(20)`, `T 点数 ~ O(3)`。
   禁止: `N_cells = 17`, 具体 SOC 百分比, 具体温度点集合。
5. **无量纲诊断量的量级**
   允许: `Hessian 条件数 ~ 1e8`, `残差 lag-1 自相关 ≳ 0.9`, `RMSE 量级 %`。
   禁止: `2.37e8`, `0.923`, `4.17%`。
6. **布尔性判断**
   允许: "收敛 / 未收敛", "命中边界 / 未命中", "残差白 / 残差非白"。
7. **残差形状的定性分类**
   允许的分类标签: `systematic_curvature | heteroscedastic | autocorrelated |
   white | other`。残差图本身不可外带。
8. **参数相对不确定度的量级**
   允许: `k_SEI_cal 相对 σ > 10%`, `R_SEI 相对 σ < 1%`。
   **禁止附任何绝对值**。
9. **物理机理的自然语言描述**
   允许: "怀疑 plating 在 pre-knee 段被激活", "SEI 瞬态未 plateau"。

## §3 绝对不可外带清单 (negative list)

凡以下类别, **无论脱敏程度如何, 一律不外带**:

1. **任何绝对数值**
   参数值、容量、电阻、温度 (K 或 °C)、SOC 百分比、时间 (s/h/月)、EFC。
   哪怕只是 "大概 4.2 Ah 左右" 这种措辞也不行 (仍然泄露量级 × 具体倍数)。
2. **残差曲线、诊断图、拟合图**
   包括 matplotlib PNG、ASCII 艺术、base64 嵌入。即使去除坐标轴标签也不可,
   因为曲线形状本身可被识别到具体 cell 批次。
3. **Cell 元信息**
   型号、供应商、批次号、生产日期、标称容量、datasheet 条目、采购渠道。
4. **数据规模的精确值**
   只给量级 (§2.4)。`N_cells = 17` 不等价于 `N_cells ~ O(10)`, 前者已经泄露。
5. **实验时间窗**
   不允许 "本月", "上周三"; 可以 "近期 RPT" / "某次 RPT"。
6. **路径中的 cell ID / fixture ID / channel ID**
   `cell_E7`, `channel_3` 全部替换为 `cell_X`, `channel_Y`。

## §4 观测笔记模板

工程师在离线区填写, 外带时**原样**贴给在线 Claude。严禁手工添加 §3 中的任何字
段, 以免绕过 `--export-public` 审查 (见 §5)。

```markdown
## 观测笔记 YYYY-MM-DD

- 脚本: scripts/fit_{...}.py @ git <hash>
- 错误码 / 退出状态:
- 数据规模量级: N_cells ~ O(?), N_RPT/cell ~ O(?), T 点数 ~ O(?)
- 症状 (定性描述, 无数值):
- 残差形状分类: [systematic_curvature | heteroscedastic |
                 autocorrelated | white | other]
- 优化器行为: [converged | stalled | boundary_hit | oscillating]
- Hessian 条件数量级:
- 已查 runbook: [错误码列表]
- 已试处置:
- 我的假设:
- 希望 Claude 协助判断的具体问题:
```

每一项都是必填; 未知写 `未知`, 不填 `N/A` 或留空 (留空会被误读为"零")。

## §5 `--export-public` 开关的语义与审计

离线区脚本若需要生成一份"可外带文本", 使用 `--export-public` 参数:

```
python scripts/<name>.py --export-public
```

### 5.1 默认机制

- 脚本打印 bundle 预览到**终端**
- 终端提示 `是否外带? [y/N]` (默认 N)
- 只有明确输入 `y` 才会把 bundle 写入 `runs/{run_id}/public_bundle_<timestamp>.md`

### 5.2 硬上限

- Bundle 文本长度 ≤ **20 行** 或 ≤ **2 KB**, 超过则脚本直接 abort 并打印超限提示
- 硬上限是 defense-in-depth: 即使有人误填入绝对数值, 总量也被限制
- 超限的合法场景不存在; 若真需要更长, 说明在尝试把数据当日志发出

### 5.3 Schema 白名单

- Bundle 只接受 §4 模板里存在的字段名
- 未在模板中的字段, 脚本**拒绝写入**并提示 "字段 X 不在 public schema"
- schema 定义与 §4 模板同源, 改模板要同步改 schema

### 5.4 文件名标记

- 所有 bundle 必须以 `_public_` 为文件名组成部分
  (如 `public_bundle_2026-04-23T10-00-00.md`)
- 便于 `git-secrets` / 企业 DLP 钩子按文件名模式做最后一道拦截

### 5.5 审计副本

- 每个 bundle 在离线区留副本到 `runs/{run_id}/`
- 副本用于事后审查 "外带了什么" 与 "什么时间外带"
- **副本不得清理**: 即使外带内容最终未被使用, 副本仍是审计轨迹

### 5.6 关于"不要求手动复制粘贴"的人因决策

早期讨论曾提议: 让工程师**手动**把 bundle 内容从脚本输出抄到 Claude 对话框,
以此"强制人眼过一遍"。**本工程否决了这一方案**。理由:

1. 重复性手工抄写反而会让工程师 desensitize, 抄到后面只机械复制, 审查性下降
2. 抄写过程中极易掺入本地粘贴板残留 (可能带有实验数据)
3. 终端 y/N 确认 + 硬上限 + Schema 白名单 + 文件名标记 + 审计副本这五层机制
   已提供比手工抄写更可靠的拦截

因此最终方案: **自动写文件 + 终端 y/N 确认 + 审计副本**。

## §6 和 Claude 对话时的措辞规范

### 例 1 — 数据契约违规 (DATA 系列)
❌ "我跑了 23 个 cell, `R_IR_mOhm` 列大概 32% 缺失, SEI 拟出来是 0.043"
✅ "`N_cells ~ O(10)`, 在 EXP-E 数据上触发 `DATA-E001`; 缺失率量级 ~30%; 已按 runbook §2 处置仍无进展, 希望讨论是否应把 R_SEI 从 free 改 fixed"

### 例 2 — 拟合前置条件不足 (FIT4A/B/C 系列)
❌ "FIT-4a 跑出来 k_SEI_cal = 3.2e-8, 看着偏小, 是不是我温度点太少了?"
✅ "触发 `FIT4A-E002`, T 点数 `~ O(2)`; 已固定 `E_a_SEI` 到文献量级; 希望讨论:单温度情形下 k_SEI_cal 相对 σ 量级约 >10% 时, 是否应放弃该参数的独立识别并改回论文默认"

### 例 3 — 环境漂移 (ENV 系列)
❌ "conda env 里 scipy 升到了 1.14.2, 跟之前的 1.11.x 差了两版, 现在 BDF 行为不一样了, 怎么办"
✅ "触发 `ENV-E001`; scipy 主次版本偏离基准 (具体版本不外带); runbook §1 处置 2 已尝试, 问题是团队需要那个新版本的另一个功能; 希望讨论是否需要在基准侧升级锁定版本"

### 例 4 — R2 顺序违规 (FIT4B-E001 等)
❌ "我在 FIT-4b 里加了 R_SEI 当自由参数, 想让 IR 更准, 结果 knee 反倒跑偏了"
✅ "触发 `FIT4B-E001` (R_SEI 被解冻); 这是故意的: 想验证是否可以用循环 IR 数据去复核日历拟合结果; 目前看起来 k_LP 的估计偏早 (量级 ~20%); 希望讨论这个实验是否有方法学上的意义, 还是应该严格遵守 R2"

### 例 5 — 求解器数值异常 (SOLVE 系列)
❌ "solve_ivp 在 t=3600s 左右 status=-1, 跑的是 CP 放电 10W 到 2.5V, SOC 初值 0.5"
✅ "触发 `SOLVE-E001`, 发生在 CP 放电收尾段 (SOC 量级低端); runbook §6 处置 1/2 已尝试; 希望讨论: 是否在低 SOC 段 CP 模式本来就不该用, 而应由 event 驱动切换到 CC/CV"

### 例 6 — 识别性警告 (IDENT 系列)
❌ "Hessian 条件数 2.37e8, k_SEI_cal 和 k_LAM_PE_cal 之间 cov 很大"
✅ "触发 `IDENT-W001`, 条件数量级 ~1e8; 特征向量指向 k_SEI_cal ↔ k_LAM_PE_cal 线性组合; SOC 覆盖仅半段; 希望讨论: 是应当 fix k_LAM_PE_cal 还是重新设计 DOE"

## §7 审计轨迹要求

每次 `--export-public` 生成 audit log 到 `runs/{run_id}/audit.log`, 一行一条,
字段用 `|` 分隔:

```
<ISO8601 timestamp> | <git commit hash> | <script name> | <bundle sha256> | <user y|N response>
```

关键点:

- Audit log 只记录**元数据**, 不包含 bundle 正文 (正文在 `public_bundle_*.md` 里已有)
- `bundle sha256` 用于事后比对 "对话里贴的和本地记录的是否同一份"
- `user y|N response` 必填; 拒绝外带也要记录 (证明机制生效)
- Audit log 不得删除, 建议与 `runs/{run_id}` 一起进入 cold storage
